# Agentic automations — tool-using headless runs through the chokepoint — design

**Date:** 2026-07-03
**Status:** approved (brainstormed + user-approved)
**Traces to:** new FR-84…FR-87 (docs/03), ADR-004 (scheduler + headless runs), ADR-005 (AXON owns its MCP server), ADR-007 (chokepoint), ADR-015 (provider routing), new ADR-017; supersedes the "headless automations deliberately do not use tools" posture in docs/08 §3 for the automations opted in here.

## Goal

Let scheduled automations run Claude **with AXON's own MCP tools** — bounded
turns, a per-run token cap enforced in real time, a read-only tool allowlist
enforced on both sides of the wire — so knowledge-digest can actually read the
week's sources instead of being told a count, and compaction can check
backlinks before distilling. Doc 08 named this extension and its precondition:
*"it would require per-turn budget enforcement before adoption."* This spec is
that adoption, precondition first.

## Background / constraints

- Today's headless shape (`internal/agent/claudecode.go buildArgs`) is
  deliberately inert: `--print --output-format json --max-turns 1 --tools ""
  --setting-sources "" --strict-mcp-config`, prompt on stdin. `agent.Request`
  has no tool/turn fields.
- The Claude CLI (verified July 2026) supports: `--mcp-config` (inline JSON),
  `--strict-mcp-config`, `--allowedTools "mcp__<server>__<tool>"`,
  `--tools ""` (drops every built-in tool), `--max-turns N` (errors when hit),
  `--output-format stream-json` (per-turn events; final event carries
  cumulative usage + `num_turns` + result), `--no-session-persistence`,
  `--setting-sources`.
- The AXON MCP stdio server (`axon mcp`) already exists with 14 tools; the
  `.mcp.json` launch shape (binary + `mcp --config X --profile Y`) is already
  generated by claudeassets. Multi-process DB access is safe: WAL +
  `busy_timeout(5000)` (both processes open their own pool).
- **`automations.<name>.budget_tokens` is currently dead at runtime** — read
  only for display; no automation sets `AgentCall.BudgetTokens`. Doc 06's
  budget pre-check is specced but unimplemented. This slice wires it.
- ADR-015: `models.synthesis` is always the Claude provider (validated) —
  and both v1 agentic automations are synthesis-tier.
- Cardinal rules: every model call through the chokepoint, ledgered on every
  path; no vault mutation that isn't wikilink-safe. The engine guarantees
  "never leave a half-edited note".

## Decisions (user-approved)

1. **Per-turn enforcement via a streaming kill-switch.** Agentic runs use
   `--output-format stream-json`; the adapter accumulates real usage per
   assistant turn and kills the process group the moment the run's token cap
   is exceeded, returning the accumulated usage with a budget error. The
   manager ledgers **real** usage for killed runs (an upgrade to the failure
   path, which today ledgers only the pre-flight estimate). `--max-turns` is
   the belt on top. *Rejected:* post-hoc-only accounting (a run could
   overshoot its cap within its allowed turns — weaker than doc 08 demands);
   `--max-budget-usd` (dollar-denominated; meaningless on subscription).
2. **Read-only tools; writes stay in Go.** Agentic allowlists draw from:
   `vault_search`, `vault_read`, `vault_links`, `knowledge_search`,
   `tokens_status`. The model researches; the automation's Go code writes
   afterwards (wikilink-safe `vault.Patch` into managed blocks, unchanged).
   Dry-run stays Authorize-only. *Deferred:* wikilink-safe write tools during
   runs — a future slice with its own dry-run story.
3. **v1 scope: knowledge-digest + compaction**, via a generic mechanism (an
   automation declares tools + turn cap in code; `automations.<name>.agentic:
   false` opts back into the current one-shot prompts, which are kept as the
   fallback path). *Deferred:* a new weekly-review automation (P4's proactive
   layer).

## Design

### Adapter (`internal/agent/claudecode.go`)

- `agent.Request` gains:
  - `Tools []string` — AXON MCP tool names (e.g. `"vault_search"`); empty =
    the existing one-shot shape, byte-for-byte unchanged.
  - `MaxTurns int` — agentic turn cap (adapter enforces a floor of 1 and a
    sane ceiling, e.g. 32).
  - `RunBudgetTokens int` — per-run total token cap for the kill-switch
    (0 = uncapped run, still turn-capped).
- `ClaudeCodeOptions` gains `MCPConfigJSON string` (the inline `--mcp-config`
  payload) and `AgenticTimeout time.Duration` (default 10 min; one-shot keeps
  the existing 120 s).
- Agentic argv (when `len(Tools) > 0`):
  `--print --output-format stream-json --max-turns <N> --tools ""
  --strict-mcp-config --mcp-config <inline JSON> --allowedTools
  mcp__axon__<t1> … --no-session-persistence --setting-sources ""
  [--model …] [--append-system-prompt …]`, prompt on stdin. No hooks are
  loaded and none are needed: with `--tools ""` there are no built-in tools
  for PreToolUse to guard, and the only callable surface is the allowlisted
  read-only MCP set.
- The MCP config JSON mirrors claudeassets' `.mcp.json` entry — the running
  binary (`os.Executable()`), `["mcp", "--config", <cfg>, "--profile",
  <profile>, "--tools", <csv of the call's allowlist>]` — composed in
  `cmd/axon/deps.go` and passed through `ClaudeCodeOptions`.
- **Streaming executor:** a new exec path pipes stdout, decodes
  newline-delimited JSON events, and:
  - accumulates `usage` from each assistant-turn event into a running total
    (input + output + cache);
  - kills the process group when `RunBudgetTokens > 0` and the total exceeds
    it, returning `(Response{Usage: accumulated}, ErrRunBudgetExceeded)`;
  - on the final `result` event returns
    `Response{Text, Model, Usage: cumulative, Turns: num_turns}`.
  - Exact stream-json event field names are verified against the CLI during
    implementation (the contract here is the behavior, not the field names);
    a `--max-turns`-exhausted run surfaces the CLI's error with partial usage.
- `agent.Response` gains `Turns int` (0 for one-shot). `agent.ErrRunBudgetExceeded`
  is a sentinel the manager can test with `errors.Is`.

### MCP server: server-side tool filtering (`internal/mcp`, `cmd/axon`)

`axon mcp` gains `--tools <csv>`: when set, `NewServer` registers **only** the
named tools. Determinism over good intentions (NFR-05): the agentic subprocess
server physically lacks every other tool, so `--allowedTools` (client-side) and
`--tools` (server-side) must both fail for anything else to be callable. Tools
that call models or fetch the network (`automations_run`, `knowledge_ingest`,
`vault_write`, `vault_patch`, `vault_move`, `daily_append`, `memory_remember`)
are never in agentic allowlists in v1.

### Chokepoint (`internal/tokens/manager.go`)

- `AgentCall` gains `Tools []string` and `MaxTurns int`.
- **Agentic requires Claude:** if `len(Tools) > 0` and the resolved provider
  is not `claude`, `Run` fails with a config-shaped error (never silently
  drops tools). In practice both v1 automations use `synthesis`, which
  ADR-015 validation pins to Claude.
- `BudgetTokens` semantics split by shape:
  - one-shot (no tools): pre-flight **input** cap, exactly as today;
  - agentic: passed through as `Request.RunBudgetTokens` — the per-run
    **total** cap the streaming kill-switch enforces. Pre-flight window
    checks (day/week, guard, downgrade-to-Claude-tiers) run unchanged on the
    initial prompt estimate.
- **Failure-path upgrade:** `recordFailure` ledgers `res.Usage` when the
  adapter returned non-zero accumulated usage (killed agentic runs, partial
  turn-capped runs), falling back to the pre-flight estimate otherwise.
  `ErrRunBudgetExceeded` is surfaced as a distinct event
  (`token.run_budget_kill`) plus the standard `:failed` ledger row with real
  numbers.
- Downgrade note: agentic calls are synthesis-tier; if the window check
  downgrades to a cheaper Claude tier, tools ride along unchanged (the tier
  affects the model string only).

### Engine + automations (`internal/automations`)

- **Wire the dead config:** `RunCtx` gains `BudgetTokens int`, populated by
  the engine from `automations.<name>.budget_tokens`. `runModel` copies it
  into `AgentCall.BudgetTokens` when the call doesn't set its own. This
  activates doc 06's budget pre-check for **all** one-shot automations too —
  a deliberate behavior change (starter budgets are generous; CHANGELOG'd).
- New helper `runAgentic(ctx, rc, call tokens.AgentCall, tools []string,
  maxTurns int)` beside `runModel`: fills `Tools`/`MaxTurns`/`BudgetTokens`,
  honors dry-run (Authorize-only, no subprocess, no tokens), and treats
  `ErrRunBudgetExceeded` like a defer (graceful degradation, surfaced in the
  run summary) rather than a hard automation failure.
- `config.Automation` gains `Agentic *bool` (`yaml:"agentic,omitempty"`):
  nil = the automation's own default (true for knowledge-digest and
  compaction), false = the preserved one-shot path.
- **knowledge-digest (agentic):** tools `knowledge_search`, `vault_read`,
  `vault_links`; ~8 turns. The prompt asks the model to survey the week's
  new sources by searching and reading them, then produce the digest with
  grounded `[[wikilinks]]` to real notes. The Go code still writes the digest
  note/managed block exactly as today.
- **compaction (agentic):** tools `vault_read`, `vault_links`; ~4 turns per
  note. The model may read the note and its backlinks before distilling, so
  summaries preserve what inbound links rely on. Archive-before-patch is
  unchanged.
- Both keep their one-shot prompts as the `agentic: false` fallback and for
  graceful degradation when the agentic run is killed/deferred: the run
  summary says which path executed.

### Prompt-injection posture (NFR-05)

Vault and source content read through tools is data. The agentic system
prompt carries the same data-not-instructions framing used elsewhere; the
enforcement, as always, is structural: read-only tools, no built-ins, no
network, no writes, capped turns, capped tokens, kill-switch. The worst a
poisoned note can do is waste a bounded number of tokens.

### Observability

- One ledger row per agentic run (cumulative usage), as today; `Turns`
  surfaced in the `token.ledger` event data and the automation run summary
  ("digest (agentic, 6 turns): …").
- `token.run_budget_kill` event on kill-switch trips.
- Dashboard needs no SPA changes (events + ledger flow through existing
  channels).

### Testing

- Adapter: table-driven argv tests (one-shot shape unchanged byte-for-byte;
  agentic shape exact); streaming executor tests against a fake subprocess
  (injectable `run`/pipe seam): cumulative usage, kill at cap (process-group
  kill observed), final-result parsing, turn-limit error with partial usage.
- Manager: agentic-requires-claude error; BudgetTokens split semantics;
  recordFailure real-usage ledgering (killed run writes accumulated numbers);
  `token.run_budget_kill` event.
- Engine/automations: budget_tokens wiring (config → RunCtx → call);
  `agentic: false` fallback selects the one-shot path; dry-run makes no
  subprocess call; digest/compaction agentic prompts include the tool
  guidance; degradation path on ErrRunBudgetExceeded.
- MCP: `--tools` filtering registers only named tools (list via the SDK
  handshake in a test).
- One gated live e2e (needs `claude` CLI + login): a 2-turn agentic run with
  `--tools vault_search` against a scratch vault, asserting a tool call
  happened and usage was ledgered. Skipped in CI.

### Docs

- **ADR-017**: "Agentic automations through the chokepoint with streaming
  per-turn budget enforcement" — records the stream-json kill-switch, the
  read-only v1 tool policy, dual client/server allowlisting, the
  budget_tokens activation, and the supersession of doc 08 §3's tool-less
  posture for opted-in automations.
- **FR-84…FR-87** in docs/03:
  - FR-84 (M): agentic runner — MCP-tooled headless runs through the
    chokepoint, Claude provider only, no built-in tools, no hooks needed.
  - FR-85 (M): per-turn budget enforcement — streaming usage accumulation,
    kill at cap, real usage ledgered on every path including kills.
  - FR-86 (M): dual tool allowlisting — `--allowedTools` client-side and
    `axon mcp --tools` server-side; read-only set in v1.
  - FR-87 (S): agentic knowledge-digest + compaction with `agentic: false`
    fallback; `automations.<name>.budget_tokens` enforced at runtime.
- docs/06 (runner section), docs/07 (budget semantics), docs/08 (§3 rewrite:
  the extension is now real; the inert shape remains the default for
  everything not opted in) updated; example config + starter comments.

## Trade-offs accepted

- Agentic runs are slower and costlier than one-shot (multiple turns, tool
  round-trips); mitigated by turn caps, the newly-live budget_tokens, the
  kill-switch, and the guard. P3's local routing already freed the cheap
  tiers, so the spend lands where the frontier model earns it.
- The stream-json event schema is a CLI surface that may evolve; isolated in
  one adapter function behind tests (same posture as parseClaudeJSON today).
- Enforcing budget_tokens changes one-shot behavior for existing installs
  (previously-dead config becomes live); starter values are generous and the
  defer path degrades gracefully.

## Out of scope

- Write tools during agentic runs (future slice; needs an MCP dry-run story).
- New automations (weekly-review belongs to the proactive layer, P4).
- `runner: inprocess` agentic (api_key SDK tool loop).
- Per-automation `model:` config override wiring (still display-only).
- Interactive/Claude Desktop changes; subagent (`--agent`) usage.
