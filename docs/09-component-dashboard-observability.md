# 09 ÔÇö Component: Dashboard & Observability

**Owns:** FR-60ÔÇĞFR-64, NFR-07.
**Goal:** See, in real time, anything relevant ÔÇö tokens, costs, runs, ingestion and the knowledge graph ÔÇö and provide an in-vault knowledge surface too.

## 1. Two surfaces

1. **AXON operational dashboard** (this component): a local web app served by the daemon, live-updated over SSE, for *system* metrics.
2. **Obsidian-native knowledge surface** (generated by `axon init`/`knowledge-reindex`): Dataview/Bases dashboard notes + Obsidian's native Graph view, for *knowledge* navigation inside the editor. Keep the plugin set lean (Dataview, Tasks, Templater, Periodic Notes, Calendar assumed).

## 2. Operational dashboard

- **Stack:** a **Vite + React + Recharts** single-page app under `web/` ÔÇö the only JavaScript in the project. Built to `web/dist` and embedded into the Go binary via `embed.FS`, then served by `internal/dashboard` at `dashboard.host:port` (default `127.0.0.1:7777`). The Go side exposes a small REST + SSE API; the SPA is a pure client of it. Localhost-bound; holds no secrets (FR-63). The server never calls Claude and never free-form writes; its **only mutations are review-queue resolutions** (ADR-020) ÔÇö `GET /api/review` + `POST /api/review/action` (accept/dismiss applied through the vault's wikilink-safe ops, guarded by a CORS-preflight-forcing JSON+header requirement) ÔÇö surfaced as the **Review tab**. Every chart's dataset exports as CSV/JSON via `GET /api/export` (FR-64). `web/` needs a Node toolchain at build time only; the distributed binary needs nothing.
- **Transport:** REST for snapshots + queries; **SSE** (`/events`) for the live stream off the in-process event bus. (WebSocket acceptable if bidirectional needs arise; SSE is sufficient and simpler.)
- **Profile-aware:** the dashboard shows the profile this installation is bound to. Personal and work are separate installations, so there is normally one daemon and one dashboard per machine (default port 7777 for both).

### Views (all live)
| View | Content | Source |
|------|---------|--------|
| **Tokens** | tokens over time; stacked by automation and by model; input/output/cache split | `token_ledger` |
| **Usage & budget** | day/week token-window gauges + guard state; on `api_key` mode also cost vs cap and projected spend | `token_ledger` + `budget_state` |
| **Runs** | timeline of automation runs; status (ok/skipped/failed/dry-run); skip reasons; duration; success rate | `runs` |
| **Ingestion** | sources/day; success vs failed/redacted; embedding queue depth; throughput | `sources` + queue |
| **Vault growth** | notes, links, words over time; inbox backlog; review-queue size | `notes`/`links` snapshots |
| **Knowledge graph** | interactive graph: nodes=notes, edges=wikilinks (+ optional similarity edges, toggle); filter by folder/tag; click ÔåÆ note metadata | `links` + `vec_chunks` neighbours |
| **Activity feed** | live structured event log; filter by kind/level | `events` (SSE) |

- **Knowledge graph rendering:** D3-force or a lightweight graph lib; cap rendered nodes with filtering for big vaults; similarity edges computed from top-N vector neighbours above a threshold (cached).
- **Data export (C):** any chart's series downloadable as CSV/JSON (FR-64).

## 3. Event model

A single in-process **event bus** is the spine: automations, ingestion, the token manager and the MCP server all emit typed events. Subscribers: the SSE endpoint (live dashboard), the `events` table (history), and `.axon/logs/*.log` (human-readable mirror). Event shape:

```ts
type AxonEvent = {
  ts: string; level: 'info'|'warn'|'error';
  // The kind union as emitted by the daemon (SSE frames use `event: <kind>`;
  // the SPA's listener list in web/src/App.jsx must mirror this):
  kind: 'automation.run'|'automation.skip'|'automation.fail'
      | 'ingest.done'|'ingest.skip'|'ingest.enrich'
      | 'ingest.embed.fail'|'ingest.embed.skip'|'ingest.review_queue.fail'
      | 'token.ledger'|'token.deny'|'token.defer'|'token.downgrade'|'token.error';
  message: string;
  data?: Record<string, unknown>;   // never contains secrets
};
```
NFR-07: any Claude call / run / ingest / error appears on the dashboard within Ôëñ5s (success criterion S4).

## 4. In-vault dashboards (generated)

Written as Markdown notes in `.axon/dashboards/` (and/or `MOCs/`), e.g.:
- **Inbox & triage** ÔÇö Dataview of `00-Inbox` + the review-queue.
- **Active projects** ÔÇö Dataview over `type: project, status: active` with last-updated and linked daily notes.
- **Recent knowledge** ÔÇö latest ingested sources with summaries.
- **Link suggestions** ÔÇö render `.axon/review-queue.md` proposals.

These let the human work entirely inside Obsidian while the operational dashboard handles the system telemetry.

## 5. Health & diagnostics
- `GET /health` ÔåÆ daemon, DB, Ollama, scheduler, last successful run per automation; plus the active embeddings provider/model/dim, the running `version`, and `latest_version`/`update_available` from the daily cached release check (never a live network call). Also carries `ask_enabled`/`capture_enabled`/`related_enabled` so the SPA hides disabled tabs.
- Powers `axon doctor` and a header status pill on the dashboard.
- `GET /api/related?path=ÔÇĞ` ÔåÆ `{related:[{path, similarity}]}`, the **Related** tab's data source and the documented loopback endpoint an Obsidian sidebar plugin can consume (R8/FR-150). **Zero model calls** ÔÇö pure vector math over the ANN seam. Gated by `dashboard.related_enabled` (default-ON) + an `X-Axon-Related` header (CORS-preflight-forcing) on top of the loopback/Host guard.

## 6. Acceptance checks
- Triggering an ingest or automation shows up in the activity feed and the relevant chart within Ôëñ5s (FR-60/FR-62/S4).
- Token chart breaks spend down by automation and model; budget gauge reflects `axon status` (FR-60).
- The knowledge graph renders nodes/edges from the vault and filters by tag/folder (FR-61).
- Dashboard binds to localhost and exposes no secrets (FR-63/NFR-05).
